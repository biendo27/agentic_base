## CI/CD Contract

This scaffold persists `ci_provider: {{ci_provider}}` in `.info/agentic.yaml` and emits exactly one CI provider contract:

- `github`: `.github/workflows/*.yml` calls the shared `tools/*.sh` contract
- `gitlab`: root `.gitlab-ci.yml` plus `.gitlab/ci/*.yml` call the same shared `tools/*.sh` contract

GitLab native validation is macOS-only by contract:

- register a project runner with `tags: [macos]`
- use a shell executor on a macOS machine
- install Xcode on that runner
- do not treat Linux runners as a replacement for iOS/native validation

CI preserves the same local evidence vocabulary and stores `{{{evidence_dir}}}` as an artifact.

## Recommended Default Gitflow

Recommended default Gitflow for teams that want one shared branch model:

- `feature/*` -> `develop`
- `release/*` -> `main`
- `hotfix/*` -> `main`
- back-merge `release/*` and `hotfix/*` into `develop` after production promotion

That guidance lives in [`docs/07-agentic-development-flow.md`](./docs/07-agentic-development-flow.md). It is a recommended human workflow, not part of `.info/agentic.yaml`.

## Release Boundary

- `./tools/release.sh` can prepare and upload Firebase, TestFlight, Play, and App Store artifacts
- credentials and signing remain human-managed
- final production store publish remains a human approval step
