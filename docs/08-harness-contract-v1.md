# 08. Harness Contract V1

## Scope

This document defines the minimum truthful contract for a harness-first Flutter repo generated by `agentic_base`.

It is a product contract first, not an implementation-detail dump. Every clause maps to a machine-readable field, generated file, script, evidence bundle, or validation gate.

## Definition

A repo may claim Harness Contract V1 only when it is a generated Flutter repo that an agent can:

- understand from a finite canonical context surface
- run through deterministic entrypoints
- verify through named gates
- evolve without guessing ownership
- release up to explicit human approval boundaries

## Required Core Clauses

| Clause | Requirement | Current Backing |
| --- | --- | --- |
| Canonical context | Repo must expose one finite human-readable context surface. | Generated `README.md`, `docs/01-07`, `AGENTS.md`, `CLAUDE.md`, and the canonical-doc list written to `.info/agentic.yaml` |
| Machine contract | Repo must expose one machine-readable source of truth. | `.info/agentic.yaml` via `AgenticConfig` and `buildAgentReadyConfigMap()` |
| Deterministic entrypoints | Setup, run, test, verify, build, release-preflight, and release commands must exist with stable names. | Generated `tools/*.sh`, Make targets, and `GeneratedProjectContract.requiredPaths` |
| Ownership boundary | Generator-owned vs human-owned surfaces must be explicit. | `agent_ready_repo_contract.dart`, generated adapters, manifest ownership sections |
| Capability boundary | Optional capabilities must extend the repo through declared seams, not hidden file drops. | Module registry, installer, startup integration generator, manifest provider mapping |
| Eval contract | Repo must define what checks prove a change or release candidate is trustworthy. | `verify.sh`, generated CI, package tests, named gate vocabulary |
| Evidence contract | Meaningful runs must emit inspectable artifacts, not only exit codes. | `tools/_common.sh`, `verify.sh`, `release-preflight.sh`, generated CI artifact uploads |
| Approval contract | Human-only checkpoints must be explicit and finite. | `.info/agentic.yaml`, `release-preflight.sh`, `release.sh`, generated docs/adapters |

## Hard Guarantees

Generated repos that claim Harness Contract V1 guarantee:

1. `README.md`, `docs/`, and thin adapters agree on the same repo story.
2. `.info/agentic.yaml` remains the single machine-readable source of truth.
3. Stable `tools/` entrypoints exist and stay generator-owned.
4. Generated validation never implies unattended production publish authority.
5. Capability and provider choices stay declarative enough for agents to inspect before editing code.
6. Verification results are summarized as named gates plus inspectable evidence, not only prose.

## Explicit Non-Goals

Harness Contract V1 does not guarantee:

- a generic cross-stack kernel
- unattended credential setup
- unattended final App Store or Play production publish
- support for every Flutter profile
- one magic quality score that hides missing evidence

## Mechanical Enforcement Map

Current enforcement already exists in several places:

- `lib/src/config/agent_ready_repo_contract.dart`
  - canonical docs list
  - thin adapter files
  - deterministic script names
  - generator-owned vs human-owned paths
- `lib/src/generators/generated_project_contract.dart`
  - required generated files
  - forbidden leftovers
  - CI provider surface validation
  - generated README, doc, and adapter validation
- generated `tools/verify.sh`
  - contract-surface checks
  - toolchain checks
  - source generation
  - static analysis
  - tests
  - app-shell smoke
  - native readiness where the host supports it
- generated `tools/release-preflight.sh` and `tools/release.sh`
  - credential checks
  - platform tool checks
  - approval-state outputs
  - explicit human warning for production publish boundaries

The implemented V1 surface adds three first-class layers beyond the older scaffold contract:

- profile and support-tier metadata in `.info/agentic.yaml`
- named eval gates plus canonical evidence bundles
- explicit approval vocabulary shared across manifest, scripts, docs, and evidence

## Contract Shape

The V1 contract stays intentionally small:

- context
- execution
- ownership
- profile plus derived support tier
- capabilities plus providers
- eval plus evidence
- approvals
- SDK/version policy

## Current Rollout Freeze

The active profile-execution rollout freezes three product-policy rules without pretending later phases are already complete:

- `subscription-commerce-app` is the canonical V1 golden path
- thin base stays separate from profile-owned default-on capability packs
- `evidence_quality` means run-evidence quality, not richer agent telemetry

[`docs/15-default-app-service-matrix.md`](./15-default-app-service-matrix.md) is the canonical reference for that freeze.

If a clause cannot fit inside that shape, it belongs in supporting docs rather than widening the manifest indefinitely.

## Legacy Migration Rule

Harness Contract V1 extends the older scaffold contract additively:

1. keep existing top-level repo metadata fields
2. keep the stable execution surface names
3. keep the human approval checkpoints
4. write the `harness` section only through generator-owned create/init/upgrade flows
5. treat legacy repos without a validated `harness` section as pre-V1 until `init` or `upgrade` repairs them honestly
