# 08. Harness Contract V1

## Scope

This document defines the minimum truthful contract for a harness-first Flutter repo generated by `agentic_base`.

It is a product contract first, not an implementation detail list. Every clause should map to a machine-readable field, generated file, script, or validation gate.

Status:

- design target for upcoming implementation waves
- not yet fully enforced by the current generator unless a clause is explicitly listed under "Current Backing" or "Mechanical Enforcement Map"

## Definition

A repo may claim Harness Contract V1 only when it is a generated Flutter repo that an agent can:

- understand from a finite canonical context surface
- run through deterministic entrypoints
- verify through named gates
- evolve without guessing ownership
- release up to explicit human approval boundaries

## Required Core Clauses

| Clause | Requirement | Current Backing |
| --- | --- | --- |
| Canonical context | Repo must expose one finite human-readable context surface. | `README.md`, generated `docs/01-06`, thin adapters |
| Machine contract | Repo must expose one machine-readable source of truth. | `.info/agentic.yaml` via `AgenticConfig` and `buildAgentReadyConfigMap()` |
| Deterministic entrypoints | Setup, run, verify, build, release-preflight, and release commands must exist with stable names. | Generated `tools/*.sh` surfaces plus `GeneratedProjectContract.requiredPaths` |
| Ownership boundary | Generator-owned vs human-owned surfaces must be explicit. | `agent_ready_repo_contract.dart`, generated adapter docs |
| Capability boundary | Optional capabilities must extend the repo through declared seams, not hidden file drops. | module registry, installer, startup integration generator |
| Eval contract | Repo must define what checks prove a change or release candidate is trustworthy. | today: `verify.sh`, generated CI, package tests; v1 extends this |
| Evidence contract | Meaningful runs must emit inspectable artifacts, not only exit codes. | partial today; V1 adds canonical evidence bundles |
| Approval contract | Human-only checkpoints must be explicit and finite. | `release-preflight.sh`, `release.sh`, adapter docs |

## Hard Guarantees

Generated repos that claim Harness Contract V1 must guarantee:

1. `README.md`, `docs/`, and thin adapters agree on the same repo story.
2. `.info/agentic.yaml` remains the single machine-readable source of truth.
3. Stable `tools/` entrypoints exist and stay generator-owned.
4. Generated validation never implies unattended production publish authority.
5. Capability and provider choices stay declarative enough for agents to inspect before editing code.
6. Verification results can be summarized as named gates plus evidence, not only prose.

## Explicit Non-Goals

Harness Contract V1 does not guarantee:

- a generic cross-stack kernel
- unattended credential setup
- unattended final App Store or Play production publish
- support for every Flutter profile
- a single magic quality score that hides missing evidence

## Mechanical Enforcement Map

Current enforcement already exists in several places:

- `lib/src/config/agent_ready_repo_contract.dart`
  - canonical docs list
  - thin adapter files
  - deterministic script names
  - generator-owned vs human-owned paths
- `lib/src/generators/generated_project_contract.dart`
  - required generated files
  - forbidden leftovers
  - CI provider surface validation
  - generated README and adapter validation
- generated `tools/verify.sh`
  - source generation
  - static analysis
  - tests
  - iOS simulator readiness on Darwin hosts
- generated `tools/release-preflight.sh` and `tools/release.sh`
  - credential checks
  - platform tool checks
  - explicit human warning for production publish boundaries

The design target adds three missing first-class surfaces:

- support tier + app profile encoding in `.info/agentic.yaml`
- named eval/evidence model
- explicit approval state machine

## Contract Shape

The V1 contract should stay small:

- context
- execution
- ownership
- profile + derived support tier
- capabilities + providers
- eval + evidence
- approvals
- SDK/version policy

If a clause cannot fit inside that shape, it should live in supporting docs rather than expand the manifest indefinitely.

## Migration Rule

The current repo already proves the scaffold contract. Harness Contract V1 should extend that contract additively:

- keep existing metadata fields
- keep existing execution surface names
- keep existing human checkpoints
- add new manifest sections and evidence outputs only when backed by generator changes and tests
